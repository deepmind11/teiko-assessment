# Cell Count Analysis: Teiknical Assessment

Analysis of immune cell population frequencies across clinical trial cohorts, with
statistical comparison of treatment responders vs. non-responders and an interactive
dashboard for data exploration.

## Quick Start

```bash
make setup      # Install dependencies
make pipeline   # Load data into SQLite, generate all outputs (CSV, plots, stats)
make dashboard  # Launch interactive dashboard at http://localhost:8501
```

Requires Python 3.10+. Tested on Python 3.12. All dependencies are listed in
`requirements.txt`.

## Part 1: Database Design

### Schema

The flat CSV is normalized into three tables following third normal form (3NF):

```
projects       subjects        enrollments              samples
--------       --------        -----------              -------
id (PK)        id (PK)         id (PK, auto)            id (PK)
           ┌── condition   ┌── subject_id (FK) ──┐  ┌── enrollment_id (FK)
           │   sex         │   project_id (FK) ──┘  │   sample_type
           │               │   treatment            │   time_from_treatment_start
           └───────────────┘   age                  │   response
                               UNIQUE(subj, proj)   │   b_cell, cd8_t_cell,
                                                    │   cd4_t_cell, nk_cell,
                                                    │   monocyte
                                                    │
                                              sample_view (VIEW)
                                              ──────────────────
                                              Flattens all four tables into one
                                              row per sample for analytics queries.
```

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY
);

CREATE TABLE subjects (
    id TEXT PRIMARY KEY,
    condition TEXT NOT NULL,
    sex TEXT NOT NULL CHECK (sex IN ('M', 'F', 'other'))
);

CREATE TABLE enrollments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    subject_id TEXT NOT NULL REFERENCES subjects(id),
    project_id TEXT NOT NULL REFERENCES projects(id),
    treatment TEXT NOT NULL,
    age INTEGER NOT NULL CHECK (age > 0),
    UNIQUE (subject_id, project_id)
);

CREATE TABLE samples (
    id TEXT PRIMARY KEY,
    enrollment_id INTEGER NOT NULL REFERENCES enrollments(id),
    sample_type TEXT NOT NULL,
    time_from_treatment_start INTEGER NOT NULL CHECK (time_from_treatment_start >= 0),
    response TEXT CHECK (response IN ('yes', 'no') OR response IS NULL),
    b_cell INTEGER NOT NULL,
    cd8_t_cell INTEGER NOT NULL,
    cd4_t_cell INTEGER NOT NULL,
    nk_cell INTEGER NOT NULL,
    monocyte INTEGER NOT NULL
);

-- Convenience view: flattens joins for analytics
CREATE VIEW sample_view AS
SELECT
    sa.id AS sample, sub.id AS subject, e.project_id AS project,
    sub.condition, e.age, sub.sex, e.treatment,
    sa.sample_type, sa.time_from_treatment_start, sa.response,
    sa.b_cell, sa.cd8_t_cell, sa.cd4_t_cell, sa.nk_cell, sa.monocyte
FROM samples sa
JOIN enrollments e ON sa.enrollment_id = e.id
JOIN subjects sub ON e.subject_id = sub.id;

-- Indexes
CREATE INDEX idx_enrollments_subject ON enrollments(subject_id);
CREATE INDEX idx_enrollments_project ON enrollments(project_id);
CREATE INDEX idx_enrollments_treatment ON enrollments(treatment);
CREATE INDEX idx_samples_enrollment ON samples(enrollment_id);
CREATE INDEX idx_samples_sample_type ON samples(sample_type);
CREATE INDEX idx_samples_time ON samples(enrollment_id, time_from_treatment_start);
```

### Design Rationale

**Why three tables instead of the flat CSV?**

The raw CSV repeats subject-level attributes (condition, age, sex) across all timepoint
rows for each subject. Normalizing into separate tables eliminates this redundancy and
enforces referential integrity.

**Why four tables instead of the flat CSV?**

The schema separates four concerns: who the person is (subjects), what project and
treatment they are assigned to (enrollments), what project exists (projects), and what
was measured (samples). This separation reflects the real-world clinical data model where
these are independent entities with different lifecycles.

**Why an `enrollments` junction table?**

In clinical trials, the same patient can be enrolled in multiple studies, potentially
receiving different treatments in each. In the current dataset, each subject appears in
exactly one project, but this is a coincidence of the data, not a structural guarantee.
The `enrollments` table models the many-to-many relationship between subjects and
projects, with `treatment` stored per-enrollment (since a patient could receive miraclib
in one trial and phauximab in another). The UNIQUE constraint on `(subject_id, project_id)`
prevents duplicate enrollments while allowing one subject across many projects.

**Why `sample_type` lives on `samples`, not `projects`:**

In this dataset, each project happens to use a single sample type (prj1=PBMC, prj2=WB,
prj3=PBMC). However, that is a coincidence of the current trial design, not a structural
constraint. A longitudinal study could collect whole blood at baseline for CBC and PBMCs
at later timepoints. Placing `sample_type` on the sample, where it physically belongs,
avoids this assumption and keeps the schema correct for multi-sample-type projects.

**Why `response` lives on `samples`, not `subjects` or `enrollments`:**

In the current data, every sample from a given subject has the same response value. It
would be tempting to store response as a subject-level or enrollment-level attribute.
However, treatment response can be assessed at each timepoint: a patient might be
classified as a non-responder at baseline and a responder at a later follow-up. Placing
`response` on `samples` keeps the schema correct for longitudinal response assessment
without requiring a migration later.

**Why `response` allows NULL:**

Healthy control subjects have no treatment, so treatment response is not applicable.
NULL correctly represents "not assessed" rather than a sentinel value. The CHECK
constraint locks `response` to exactly three states: `'yes'`, `'no'`, or `NULL`.

**Why `sample_view`?**

The four-table normalized schema requires multi-table JOINs for every analytical query.
Rather than embedding that join logic in every Python script, a `sample_view` flattens
the schema back into one row per sample. All analytics code (pipeline.py, dashboard.py)
reads from `sample_view`, so the normalization is invisible to data consumers. This gives
us the integrity benefits of normalization with the query simplicity of a flat table.

### Assumptions and Deliberate Constraints

The schema makes several deliberate choices about what to constrain and what to leave
open, based on observing the current data while keeping future flexibility in mind:

| Column | Table | Constraint | Rationale |
|--------|-------|-----------|-----------|
| `response` | samples | CHECK ('yes', 'no', NULL) | Binary treatment outcome is a stable domain concept. NULL for untreated/healthy subjects. Placed on samples (not subjects or enrollments) because response can be assessed at each timepoint; in this dataset it is constant per subject, but that is not guaranteed in future trials. |
| `sex` | subjects | CHECK ('M', 'F', 'other') | Covers the current data (M/F only) plus a third category for clinical data standards that allow unreported or non-binary values. |
| `age` | enrollments | INTEGER, CHECK > 0 | Age is recorded at enrollment, not as a permanent subject attribute, because the same patient enrolling in a second trial years later will have a different age. All ages in this dataset are whole numbers (30-79 range). A case exists for REAL (pediatric trials report fractional years), but INTEGER is appropriate for this adult oncology cohort. |
| `condition` | subjects | No CHECK | In the current data, each subject has one condition. A production platform would use a `subject_conditions` junction table to support comorbidities and evolving diagnoses. Kept on subjects here because the dataset and the analytical queries (filter by single condition) don't require it. No CHECK constraint because new indications should not require schema migrations. |
| `treatment` | enrollments | No CHECK | Treatment is per-enrollment, not per-subject, because the same patient can receive different drugs in different trials. No CHECK constraint because new drugs are the entire point of clinical trials. |
| `subject-project` | enrollments | UNIQUE (subject_id, project_id) | Prevents duplicate enrollments while allowing one subject across many projects. In the current data, each subject has exactly one enrollment (3,500 subjects, 3,500 enrollments). |
| `sample_type` | samples | No CHECK | Currently PBMC and WB, but new sample types (tissue biopsy, CSF, bone marrow) should not require schema changes. |
| `time_from_treatment_start` | samples | CHECK >= 0 | The current data uses days 0, 7, and 14, but different trial protocols sample at different intervals (weekly, monthly, hourly). Only non-negative values are enforced. |

For a production platform, `condition`, `treatment`, and `sample_type` would be
normalized into reference/lookup tables with metadata (ICD-10 codes for conditions, drug
class and mechanism for treatments). This enables consistent terminology across projects
and supports metadata-driven filtering. For this dataset, leaving the columns unconstrained
achieves flexibility without the overhead of reference tables with only 2-3 rows each.

### Cell Population Columns: Wide vs. Long Format

The five cell populations (b_cell, cd8_t_cell, cd4_t_cell, nk_cell, monocyte) are stored
as columns on the `samples` table rather than as rows in a separate measurements table
(the EAV/entity-attribute-value pattern).

This is a deliberate trade-off:

- **Wide format (current)** is optimal for a fixed panel of 5 populations. Queries are
  natural (`SELECT b_cell, cd4_t_cell FROM samples`), relative frequency calculations
  are simple arithmetic across columns, and NOT NULL constraints guarantee completeness
  (every sample has all 5 counts).

- **EAV format** (a `measurements` table with `sample_id`, `population`, `count`) would
  be necessary for a platform supporting variable panels. A 40-marker CyTOF panel or a
  spectral flow panel with 25+ markers cannot be modeled as 40 columns. The EAV pattern
  lets new populations be added as data rather than schema changes.

For this 5-population dataset, the wide format is the right choice. The trade-off tips
toward EAV at roughly 15-20+ populations, or when different projects measure different
subsets of populations.

### Indexes

Six indexes support the analytical query patterns used by `sample_view` and Parts 3-4:

- **`idx_enrollments_subject`** on `enrollments(subject_id)`: supports the join from
  enrollments to subjects.
- **`idx_enrollments_project`** on `enrollments(project_id)`: supports project-level
  filtering and aggregations.
- **`idx_enrollments_treatment`** on `enrollments(treatment)`: supports treatment-based
  cohort selection.
- **`idx_samples_enrollment`** on `samples(enrollment_id)`: supports the join from
  samples to enrollments.
- **`idx_samples_sample_type`** on `samples(sample_type)`: supports PBMC/WB filtering.
- **`idx_samples_time`** on `samples(enrollment_id, time_from_treatment_start)`:
  composite index supporting both the enrollment join and baseline filtering (Part 4).

At the current scale (10,500 samples, 3,500 subjects), SQLite handles queries in
single-digit milliseconds even without indexes. These indexes become meaningful at
tens of thousands of samples with concurrent dashboard users.

### Scaling to Hundreds of Projects and Thousands of Samples

1. **Query efficiency.** The `sample_view` pre-joins all four tables, and the underlying
   indexes on enrollment and sample foreign keys keep join performance constant as data
   grows. Cohort filtering (condition + treatment) uses indexed lookups on the smaller
   enrollments and subjects tables.

2. **No redundant storage.** Subject demographics are stored once per person, not once per
   sample per timepoint. Enrollment metadata (project + treatment) is stored once per
   subject-project pair. For this dataset, that eliminates 3x repetition of subject
   attributes. With more timepoints or assay types, the savings grow proportionally.

3. **Cross-project subjects.** The enrollments junction table supports the common clinical
   trial pattern where the same patient participates in multiple studies. A subject's
   demographics remain consistent across projects while their treatment assignment varies
   per enrollment.

4. **Flexible analytics.** The normalized structure supports responder vs. non-responder
   comparisons, longitudinal analyses, cross-project cohort studies, and demographic
   breakdowns, all through `sample_view` without writing JOINs in analytics code.

5. **Data integrity.** Foreign keys ensure every sample references a valid enrollment, and
   every enrollment references valid subject and project records. The UNIQUE constraint
   on `(subject_id, project_id)` prevents duplicate enrollments. CHECK constraints
   enforce domain rules where the value space is stable (response, sex, age range).

6. **Extensibility.** New projects, conditions, treatments, and sample types can be loaded
   without schema changes. Adding new cell populations requires an ALTER TABLE (or a
   migration to EAV if the population count grows beyond what columns can support).

## Part 2: Relative Frequency Summary

For each of the 10,500 samples, the pipeline computes the total cell count (sum of all
five populations) and the relative frequency of each population as a percentage of that
total. The output is a long-format table with 52,500 rows (10,500 samples x 5
populations).

Output file: `outputs/summary.csv`

| Column | Description |
|--------|-------------|
| sample | Sample ID from cell-count.csv |
| total_count | Sum of b_cell + cd8_t_cell + cd4_t_cell + nk_cell + monocyte |
| population | Cell population name (e.g., b_cell, cd4_t_cell) |
| count | Raw cell count for this population |
| percentage | 100 x count / total_count |

## Part 3: Statistical Analysis

### Cohort

Melanoma patients treated with miraclib, PBMC samples only.

- **331 responder subjects** (993 samples across 3 timepoints)
- **325 non-responder subjects** (975 samples across 3 timepoints)

### Handling Repeated Measures

Each subject has 3 samples (day 0, 7, 14). These are not independent measurements:
the same person measured at different times will produce correlated values. Using all
1,968 samples as if they were independent would be pseudo-replication, inflating
significance.

To quantify this, I measured how much each subject's cell percentages vary across
their 3 timepoints. For each of the 656 subjects, I calculated the standard deviation
of their 3 values, then averaged that SD across all subjects. I also computed the
standard deviation across all 1,968 samples (ignoring which subject they came from):

| Population | SD Across a Subject's 3 Timepoints (avg) | SD Across All Samples |
|------------|------------------------------------------|-----------------------|
| B Cell | 2.84% | 3.22% |
| CD8 T Cell | 4.17% | 4.67% |
| CD4 T Cell | 4.50% | 5.03% |
| NK Cell | 3.62% | 3.96% |
| Monocyte | 3.84% | 4.34% |

A single subject's values vary almost as much across their 3 timepoints as the
variation across all samples. The 3 measurements from one person are not 3
copies of the same number; they carry substantial fluctuation. Treating them as
independent observations would inflate the effective sample size and make p-values
artificially small.

**Fix:** I average each subject's 3 timepoint values into a single value per cell
population, then run the test on 331 vs 325 independent observations.

### Choosing the Statistical Test

I plotted the distribution of each cell population's relative frequency and on visual
inspection they appeared approximately normal (symmetric, bell-shaped). To confirm
this, I ran the Kolmogorov-Smirnov (K-S) normality test on each population for both
the responder and non-responder groups:

| Population | K-S p (Responders) | K-S p (Non-Responders) | Normal? |
|------------|--------------------|-----------------------|---------|
| B Cell | 0.3522 | 0.5397 | Yes |
| CD8 T Cell | 0.9732 | 0.9656 | Yes |
| CD4 T Cell | 0.3023 | 0.9275 | Yes |
| NK Cell | 0.7924 | 0.8706 | Yes |
| Monocyte | 0.5906 | 0.7604 | Yes |

All K-S p-values are well above 0.05, confirming that the data is normally distributed
across all populations and groups. Because the normality assumption holds, I use
Welch's t-test, which has more statistical power than the non-parametric Mann-Whitney U
test. Welch's t-test does not assume equal variances between groups.

The interactive dashboard also provides the option of running Mann-Whitney U for
comparison. With Mann-Whitney U, CD4 T Cell narrowly misses significance after FDR
correction (BH-adjusted p = 0.062). With Welch's t-test, the same signal reaches
significance (BH-adjusted p = 0.023), because the test extracts more information from
normally distributed data.

### Overall Results (Per-Subject Averaged)

Welch's t-test (two-sided) on per-subject averaged relative frequencies, with
Benjamini-Hochberg FDR correction across the 5 cell populations:

| Population | Resp. Mean | Non-Resp. Mean | p-value | BH-adjusted p | Significant |
|------------|-----------|---------------|---------|---------------|-------------|
| CD4 T Cell | 30.54% | 29.90% | 0.0045 | 0.023 | **Yes** |
| B Cell | 9.80% | 10.00% | 0.1627 | 0.274 | No |
| NK Cell | 14.84% | 15.07% | 0.1644 | 0.274 | No |
| Monocyte | 19.94% | 20.08% | 0.4524 | 0.566 | No |
| CD8 T Cell | 24.88% | 24.94% | 0.7666 | 0.767 | No |

CD4 T Cell is the only population that reaches significance after FDR correction
(BH-adjusted p = 0.023). Responders have a higher mean CD4 T cell frequency than
non-responders. To understand when this difference emerges, I examined each timepoint
independently.

### Per-Timepoint Analysis

Each timepoint provides ~331 vs ~325 fully independent observations (one per subject).
K-S normality tests confirm normal distributions at each timepoint (all p > 0.05).

**Day 0 (baseline):**

| Population | Resp. Mean | Non-Resp. Mean | p-value | BH-adjusted p |
|------------|-----------|---------------|---------|---------------|
| B Cell | 10.09% | 9.77% | 0.2115 | 0.845 |
| Monocyte | 19.90% | 20.22% | 0.3379 | 0.845 |
| CD8 T Cell | 24.82% | 25.00% | 0.6194 | 0.875 |
| CD4 T Cell | 29.98% | 29.83% | 0.7085 | 0.875 |
| NK Cell | 15.22% | 15.17% | 0.8751 | 0.875 |

No significant differences at baseline. Responders and non-responders start from
the same immunological baseline, which is expected.

**Day 7:**

| Population | Resp. Mean | Non-Resp. Mean | p-value | BH-adjusted p |
|------------|-----------|---------------|---------|---------------|
| CD4 T Cell | 30.75% | 29.72% | 0.0100 | **0.050** |
| B Cell | 9.75% | 10.11% | 0.1498 | 0.271 |
| NK Cell | 14.66% | 15.09% | 0.1626 | 0.271 |
| Monocyte | 20.05% | 20.18% | 0.6852 | 0.772 |
| CD8 T Cell | 24.80% | 24.90% | 0.7720 | 0.772 |

CD4 T Cell is significant at day 7 (BH-adjusted p = 0.050). Responders show
elevated CD4 T cell frequencies one week into treatment.

**Day 14:**

| Population | Resp. Mean | Non-Resp. Mean | p-value | BH-adjusted p |
|------------|-----------|---------------|---------|---------------|
| B Cell | 9.56% | 10.11% | 0.0310 | 0.151 |
| CD4 T Cell | 30.89% | 30.16% | 0.0603 | 0.151 |
| NK Cell | 14.65% | 14.96% | 0.3129 | 0.522 |
| CD8 T Cell | 25.03% | 24.93% | 0.7880 | 0.936 |
| Monocyte | 19.87% | 19.85% | 0.9360 | 0.936 |

B Cell shows the strongest signal at day 14 (uncorrected p = 0.031) but does not
survive FDR correction. CD4 T Cell remains a trend (uncorrected p = 0.060).

### Interpretation

The per-timepoint analysis reveals a temporal pattern in the immune response:

- **Day 0:** No differences. Both groups are immunologically equivalent at baseline.
- **Day 7:** CD4 T cells diverge first (BH-adjusted p = 0.050), consistent with their
  role in early adaptive immune activation.
- **Day 14:** B cells begin to diverge (uncorrected p = 0.031), potentially reflecting
  downstream effects of the CD4 T cell response on B cell dynamics.

This temporal progression supports a real biological effect rather than a statistical
artifact: the signal is absent at baseline (when treatment has not yet acted) and
emerges progressively at later timepoints. The sequence is also consistent with known
biology: CD4 T cells are known to provide help that drives B cell activation and
maturation, so it is plausible that CD4 T cell changes at day 7 lead to downstream
B cell changes by day 14.

## Code Design

The project separates concerns into three scripts:

- **`load_data.py`** handles data management only: it reads the flat CSV, creates the
  normalized SQLite schema, and loads all rows. It uses the standard library (`csv`,
  `sqlite3`) with no external dependencies.

- **`pipeline.py`** handles all analysis: it queries the database using SQL JOINs
  (demonstrating that the schema supports the required analytics), computes relative
  frequencies, runs statistical tests, and generates static outputs (CSV files and
  plot images).

- **`dashboard.py`** handles presentation only: it reads the database and the output
  files generated by the pipeline, then renders an interactive Streamlit application.

This separation means the pipeline can run headlessly in CI/CD or Codespaces
(`make pipeline`), and the dashboard can be started independently afterward
(`make dashboard`).
